# Proposal: crash-resilience — model-free supervisor that resumes a dead agent + a standing successor handoff (#54)

> The canonical design doc (ADR + PR description). Reviewed by `--scaffold` before build. Lands on main.

## Problem

An autonomous research run has three ways to fail, and we cover only two of them. The agent can go *idle* (parked, waiting on nothing) — caught by the self-wake cron `run-experiment` arms. The GPU pod can be *orphaned* (still billing after the run is done or abandoned) — caught by the model-free reaper `gpu-job/watchdog.sh`. The third leg is the agent process **itself dying or wedging mid-run**, and nothing reliably catches it. The honest reason is mechanical: anything that could *notice* the agent is gone and resume it must not need the same Anthropic API that just went down — a heartbeat cron driven by a model is useless during an API outage, and Claude Code's built-in SDK retry is only a few attempts and not configurable, so a longer blip kills the session. When that happens with a 2×H200 pod attached, the pod bills for hours with no brain to collect the result or tear it down (the 2026-06-13 incident: an intermittent Anthropic API error inside Claude Code stranded a `$8.78/hr` pod for ~3h). This is the boundary #52 already named: the cron survives the agent going idle, but not the agent dying.

The fix has to be a **dumb, model-free supervisor** that runs on the always-on box, makes zero model calls (so it survives a total API outage), and does two jobs: relaunch a dead/wedged agent so the run resumes where it left off, and reap any pod past its cost cap. The subtlety this design exists to resolve is that **a non-trivial piece of this already exists and partly works** — so the deliverable is not "build a supervisor from scratch," it's pinning down the *residual* gaps the existing pieces miss, deciding what is product scaffold vs instance machinery, and writing the resume contract the agent must hold up its end of.

## What already exists (and why this is design-first, not build-first)

This box already runs a non-LLM respawn loop per fleet session:

- **`claude-pane-loop.sh`** (instance) runs `claude` in a `while` loop inside its tmux pane. A non-zero exit with no deliberate-restart flag is treated as a crash and **auto-relaunched with `--continue`** (same conversation, from the on-disk session JSONL), rate-limited so a crash-storm can't spin forever (>5 crashes in 120s → give up).
- **`restart-self.sh`** (instance) lets a session restart itself `continue` / `fresh` / `handoff`; the `handoff` mode requires a `TEMP.md` "will" to exist before it drops context.
- **`relaunch-session.sh handoff`** (instance) is the actual seed primitive: it constructs the "Read `<dir>/TEMP.md` first" successor seed for a detached fresh relaunch. (`new-claude.sh` only *echoes a note* that a predecessor `TEMP.md` exists — it does not itself seed — so the supervisor's fresh-successor branch must invoke `relaunch-session.sh <name> <dir> handoff`, not `new-claude.sh`.)
- **`gpu-job/watchdog.sh`** (product) reaps one pod after a grace period unless a future-dated `/workspace/.keepalive_until_utc` is present.

So three of the issue's four asks are *partly* met already. The design's real content is the gap analysis: what the existing pieces do **not** cover, and which of those gaps are worth closing, in which layer.

### The residual gaps (what the existing pieces miss)

1. **Wedge, not just crash.** `claude-pane-loop.sh` only acts on a process *exit* (a non-zero return code). An agent that is *hung* — process alive, event loop stuck on a wedged API call or a deadlocked tool — never exits, so the loop never relaunches it. There is no liveness probe that distinguishes "working" from "wedged," and `claude-pane-loop.sh` runs a single `--remote-control` process that exposes no foreground-tool/status interface to build one from. This is the harder half — and because a wrong probe restarts *healthy quiet work*, this design **defers it to its own `needs-design` pass** rather than shipping a heuristic. The crash leg and the can't-resume-in-place leg (gaps below) are the ones closed now.
2. **No box-level pod reaper independent of the run, and no ownership source one could safely use.** `gpu-job/watchdog.sh` is armed *per run, by the agent*, and scoped to one pod id. If the agent dies before arming it (or the run's brief never armed it correctly, as #52 found a subagent doing), no reaper exists. But a box-level reaper can't simply "list all pods and delete the idle ones" — `gpu-job`'s standing rule is **"Teardown is pod-id-scoped. Never blanket-delete 'idle' pods"** (a blanket sweep would kill a live peer's compute). There is no **trusted lease/ownership record** the agent writes *at acquire time* that a model-free sweep could safely act on. Closing this gap is therefore two things, not one: a lease written atomically when a pod is acquired, and a sweep that only ever reaps **expired, registered** leases.
3. **The successor handoff is not standing.** `TEMP.md` today is written only at a deliberate `handoff` restart (and `run-experiment` mentions it only for block-prone content). For the deaths that **cannot resume in place** — a usage-policy block that kills the thread's ability to even write its own will, or a corrupted session JSONL — there is no *always-current* handoff for a fresh successor to pick up. `--continue` recovers the same session; it does not help when the same session is unrecoverable.
4. **No documented resume contract.** The agent has to hold up its end for `--continue` to actually resume *useful* work (checkpoint state to disk, not only in conversation; keep the standing handoff current; never leave a pod un-reaped behind an in-conversation-only note). None of this is written down as a product expectation, so it degrades silently — exactly the #52 failure shape.

## Approach

Split the work along the **product vs instance** seam (AGENTS.md "Two orthogonal cuts"), because the supervisor is half generic contract and half this-box wiring. The design PR lands this decomposition; the `ready` children implement each piece.

### A. The resume contract (product — `experiment-lifecycle` + `gpu-job` docs)

Write down, as a product expectation, what makes a run *resumable by a model-free supervisor* — the contract the agent must satisfy so that a blind relaunch actually recovers the run:

- **Checkpoint to disk, not only to conversation.** Run state a successor needs (what's launched, pod ids, what's been collected, the pre-registered decision rules) lives in the run's artifact dir / `START.md` / ledger, not only in the conversation. `--continue` replays the conversation, but a *fresh* successor only has the disk.
- **The standing successor handoff.** Promote `TEMP.md` from a block-only artifact to an **always-current** successor handoff, refreshed at checkpoints (pointers only — pod ids, artifact paths, the active look-again deadline, next action; never trigger-prone prose). This is the input the supervisor's fresh-successor branch feeds to `relaunch-session.sh handoff` when same-session resume is impossible.
- **Never leave a pod behind an in-conversation-only note.** A pod's existence and its cost-cap deadline must be on disk (the keepalive contract + the standing handoff), so the reaper can find it without the agent.

The *resume-contract prose* (checkpoint-to-disk, the standing-handoff discipline) is **docs/protocol**, mirroring how #52 pinned the executor substrate without building a detector. But the **run-supervision record is machine-consumed state**, so it is **NOT docs-only**: child 1 also ships a small **product helper + path/schema** for it (a `create/update/stop/close/is-desired-active` API and a defined record path), so a zero-context agent and the instance wrappers (`claude-pane-loop.sh`, the stop helpers) read/write it through one implementation rather than re-deriving atomic-write semantics from prose. This record is the precondition that makes the instance supervisor's blind relaunch safe.

### B. The pod-lease + model-free supervisor (canonical homes fixed here)

The reaper's correctness hinges on a **trusted lease**, so this design fixes the lease shape and its home rather than deferring them (the prior draft deferred the module/interface boundary, which contradicts a `ready` child).

**Two records, two owners (the seam fixed here).** Pod **deletion** is `gpu-job`'s domain; run/session **relaunch policy** is `experiment-lifecycle`'s. So this design uses **two linked records**, not one:

- a **pod lease** (`gpu-job`) — strictly **deletion-scoped**: what does this pod cost, when does it expire, how is it deleted. The sweep reads only this.
- a **run-supervision record** (`experiment-lifecycle`) — **relaunch-scoped**: is this run desired-active, is there a deliberate-stop marker, what's the `handoff_path`. It **links to** one or more pod leases (a run may hold several pods). The relaunch supervisor reads this; it never reaches into `gpu-job` policy. Its lifecycle mirrors the lease's and is equally fail-closed: **create** (desired-active, at run start), **update** (`handoff_path` / linked pod-ids at checkpoints), **stop** (a deliberate `/quit`/kill writes the stop marker), and **close** (run finished → record marked **inactive**). All writes are atomic. The critical ordering: a finished or stopped run must **clear `desired-active` BEFORE the supervisor could observe its session gone**, or the supervisor would resurrect a run that's legitimately done — so child 1 adds a **blocking close-checklist gate** ("run-supervision record cleared/inactive") right beside the existing "self-wake/watchdog cleared" teardown gate.

This keeps `gpu-job` from becoming the home for relaunch policy it doesn't own, and lets the reaper and the relauncher evolve independently.

**B.0 — The pod lease + its fail-closed lifecycle API (product — `gpu-job`, the load-bearing piece).** Because the lease is what gives the sweep *deletion authority*, the design must define the full lease lifecycle, not just the record — a reaper that can delete on an expired lease but has no defined way for a live long run to *refresh* that lease would kill the live run. So `gpu-job` gains a **fail-closed lease API** with four operations:

- **create — THREE-PHASE (intent → provisional → enriched), so there is NO un-leased billing window.** A pod starts billing the instant REST `deploy()` returns its id, and there is a (tiny but real) window between pod creation and the provisional-lease write where a crash would leave an un-leased, billing pod the sweep can only *report*. So acquire writes a **pre-deploy acquisition-intent record FIRST** — a unique deploy **nonce/name** + the durable key reference + a short default expiry — *before* calling `deploy()`, and passes that nonce as the pod's name/tag. (1) **intent** (pre-deploy): the sweep can now match an otherwise-unknown pod to a pending intent **by nonce/name** and reap it on the intent's expiry, so even a pod created-but-id-never-returned is covered. (2) **provisional** (immediately after `deploy()` returns the id): bind the real pod id to the intent. (3) **enriched** (after `wait_ssh`): add the SSH endpoint + the run's real expiry. If the flow dies at any phase, the record is already near-expiry, so the sweep reaps the half-born pod instead of letting it bill forever. This is what lets the lease honestly claim to close the orphan-before-watchdog hole — the nonce intent removes the last gap.
  - **Write-failure path (fail CLOSED, not just loud).** If the provisional-lease write itself fails *after* the pod is REST-created, "fail loudly" isn't enough — the pod is already billing with no lease. So acquire must **synchronously attempt `DELETE` with the deploying key** and exit with explicit evidence: either the pod was deleted, or an **emergency unreaped-pod record** is written (the one place the sweep/human must look) — never a silent billing orphan.
  - **Durable key reference, not just a name (so the model-free sweep can actually delete).** The lease records a key reference the *standing sweep* can resolve on its own (e.g. the `~/.config/gpu-job/` key name the reaper reads), validated at lease-creation — not merely the name of a key that exists only in the launching shell's env. A lease the sweep can't authenticate against is a lease it can't act on.
- **enrich / refresh (during the run).** Enrich fills the SSH endpoint + the run's real expiry once known; a long run then extends `expiry` — the controller-side generalization of today's pod-side `keepalive_until_utc` refresh (lifted off the pod so it survives the pod and the agent). The *run-supervision record* (not the lease) records the active expiry so a successor keeps refreshing.
- **close — ONLY after verified provider-side deletion.** Closing the lease *before* the pod is confirmed gone turns a failed teardown into an **unknown** pod the sweep will only *report* (never delete) — a billing orphan. So `close` is allowed **only after control-plane deletion is verified** (the same "verify on the deploying account's control plane that the unit is actually gone" rule `run-experiment` already requires). A teardown whose delete didn't verify leaves the lease **pending/expired** so the standing sweep retries it — close is the *consequence* of confirmed deletion, not a substitute for it.
- **expiry is the only deletion trigger.** The sweep deletes a pod **only** when its registered lease is *past expiry* — never on "looks idle." A run that's alive keeps its lease fresh; a dead run's lease lapses and the pod is reaped.

**Lease fields stay deletion-scoped (right seam).** The lease carries only what *deletion* needs — **pod id (or the pre-deploy nonce), the durable reaper-resolvable key reference, expiry/cost-cap, and — once enriched — the SSH endpoint** (deletion-relevant because the migration rule's keepalive read needs it) — with safe defaults, so a *standalone* `gpu-job` caller gets a valid, reapable lease with no run context. Anything run-shaped (owner / run-artifact path, the `handoff_path`, desired-active state) lives in the **run-supervision record** (`experiment-lifecycle`), supplied by the caller — in practice a `run-experiment` wrapper — and linked to the lease by pod id. So the lease is fail-closed for *everyone* (always reapable) without forcing experiment-shaped fields onto a generic tool, and `gpu-job` never holds relaunch policy.

Canonical home: a `gpu-job` lease registry — generic schema + the intent/provisional/enrich/refresh/close helpers in `plugins/gpu-job/skills/gpu-job/scripts/`, instance registry path under `~/.config/gpu-job/`. This fixes the "agent died before arming the watchdog" hole (the intent is written *before* deploy and the provisional lease at pod creation, not later by the agent) and gives the sweep a fail-closed authority to act on.

**B.1 — Box-level pod reap (instance sweep over the product lease).** A standing model-free sweep lists pods via the RunPod REST API and `curl -X DELETE` **only leases that are registered AND expired**; an **unknown** pod (no lease) is **reported, never deleted** — preserving `gpu-job`'s "never blanket-delete idle pods" rule. This is the always-on counterpart to the per-pod `watchdog.sh`: it catches the pod whose run never armed a watchdog, without the blanket-delete hazard. Generic decision logic (reap-expired-registered-only, report-unknown) is a `gpu-job` reference; the systemd/bash wiring + RunPod key are instance.

**Migration / back-compat (so the switch to controller-side expiry can't reap a healthy long job).** Today a long job stays alive by refreshing the **pod-side** `keepalive_until_utc`; the new authority is the **controller-side** lease expiry. The sweep must not delete a healthy old-style job mid-transition. So during migration the sweep, before deleting an expired lease, **also honors a future-dated pod-side `keepalive_until_utc`** (the existing `watchdog.sh` contract, read over the lease's enriched SSH endpoint) — a pod is reaped only when **both** the controller lease is past expiry **and** no future pod-side keepalive is present. **Fail-safe when the keepalive can't be read** (no SSH endpoint yet — still provisional — or the read fails): the sweep falls back to **lease-expiry as the sole authority** (the controller lease IS the authority; an unreadable pod-side file never *protects* a pod past its lease expiry, so the fail-safe is to reap on expiry, never to skip). Equivalently, the lease carries a refresh-contract version and the strict controller-only rule applies only to leases created under the new version. Either way, the **standalone `gpu-job` refresh path is a point-of-need requirement** (the `SKILL.md` long-job section says how to extend the controller lease, not only the pod-side file), so a non-experiment caller isn't silently reaped after the switch.

**B.2 — Relaunch supervisor (instance, scoped to what is reliably model-free).** A standing, model-free supervisor on the always-on box (systemd timer or a `setsid` bash loop — **zero model calls**) that relaunches a session whose failure can be detected *without a model and without false positives*. We deliberately split this into a `ready` first cut and a deferred harder half, because a fully reliable model-free "the root agent is healthily inside a long model turn" signal is **not yet pinned** — and a probe that gets it wrong restarts healthy quiet work.

**Supervise desired-state, not "any missing session" (avoid resurrecting a deliberate quit).** A session that was intentionally `/quit` or manually killed must NOT be resurrected (`claude-pane-loop.sh` already treats a clean exit-0 as "really exit"). So the supervisor acts only on a session that is **desired-active AND has no deliberate-stop marker** — desired-state, not mere presence. Both live in the **run-supervision record** (B.0's `experiment-lifecycle` record, NOT the pod lease): it marks the run desired-active and carries the deliberate-stop marker the supervisor honors. Outside that set, a gone session is a no-op.

Within that supervised set, the two unambiguous failure signatures (reliable now, the `ready` first cut):

- **Process exit / crash.** A non-zero `claude` exit (already what `claude-pane-loop.sh` catches) — extended only so the supervisor *also* covers the cases the in-pane loop can't: the whole tmux session/pane gone, or the box-level wrapper itself dead — *for a registered, non-stopped run*.
- **Explicit agent-declared stuck-or-blocked.** The agent (or a `StopFailure` hook) writes a small on-disk *needs-relaunch* marker — a positive, model-free signal that recovery is wanted (the policy-block / can't-resume-in-place case), which the supervisor acts on by the launch-successor path.

Both feed the same relaunch decision, stated in **substrate-neutral product terms** so the contract isn't tied to this box's scripts: `resume_same_session()` (resume in place — implemented here by `--continue`, the path `claude-pane-loop.sh` owns) preferred, else `launch_successor(handoff_path)` (a fresh successor pointed at the standing handoff) when same-session resume is impossible (corrupted JSONL, policy block). The **`handoff_path` is bound explicitly** — carried in the run's lease / needs-relaunch marker — rather than assumed to be `<workdir>/TEMP.md`, so the artifact dir and the launcher's workdir need not coincide. The instance mapping (`resume_same_session` → `--continue`; `launch_successor` → `relaunch-session.sh <name> <dir> handoff`, adapted to accept the bound `handoff_path`) lives in instance guidance, not the product doc. The supervisor carries a **single-writer kill/resume lock** so it never double-relaunches or races the in-pane loop, is **idempotent**, and keeps the loop's crash-storm cap.

**Deferred (a `needs-design` follow-up, NOT this design's `ready` set): silent-wedge detection.** Catching an agent that is *alive but hung* with no exit and no marker needs a substrate-specific root-agent activity signal that doesn't yet exist (`claude-pane-loop.sh` runs a single `--remote-control` process exposing no foreground-tool/status interface, so transcript-mtime + child-process heuristics can't yet distinguish a long healthy model turn from a wedge). Rather than ship a heuristic that risks restarting healthy work, this design **names the open problem** and routes it to its own design pass. The first cut still closes the crash leg and the can't-resume leg — the two that caused the 2026-06-13 cost incident — today.

**Canonical-home decision (fixed here, not deferred).** The **pod-lease schema + lifecycle API + reaper decision logic** are `gpu-job` (pod deletion is its domain). The **run-supervision record (desired-active + stop marker + `handoff_path`, linked to lease pod-ids) + the relaunch decision tree (`resume_same_session` / `launch_successor(handoff_path)`, substrate-neutral) + the standing-handoff contract** are an `experiment-lifecycle` / `run-experiment` reference (executor liveness is that layer's domain). The two records link by pod id; neither owns the other's policy. All *instance specifics* — the concrete command mapping (`--continue`, `relaunch-session.sh`), session names, paths, RunPod key, systemd unit, `claude-pane-loop.sh` — stay under `~/.config/` / instance scripts per the releasability rule (the product doc names the *operations*, the instance names the *commands*). The `ready` children below are `ready` because every product interface boundary they need is named here; the one genuinely-unsolved piece (silent-wedge detection) is split out as a `needs-design` child, not smuggled into a `ready` one.

### C. The cheap quick-wins (instance settings — fold in immediately)

Independent of the supervisor: bump `API_TIMEOUT_MS` and set `API_FORCE_IDLE_TIMEOUT="0"` in the instance settings so a *short* API blip is ridden out in-process rather than killing the session at all. A `StopFailure` hook can *fire* on an API error (kick the supervisor / push-notify) but **cannot itself resume** the session — documented as a notifier, not a fix, so it isn't mistaken for the recovery path.

### Decomposition into `ready` children

This design PR (doc-only) spawns these implementable units, each a normal single-phase ship-change:

1. **Resume contract + run-supervision record in the product skills + templates** (A) — `run-experiment` / `design-experiment`: the checkpoint-to-disk requirement + the standing always-current `TEMP.md` handoff (promoting it from block-only) + the **run-supervision record** schema (desired-active, deliberate-stop marker, `handoff_path`, linked lease pod-ids). Crucially this includes the **zero-context executor's point-of-need gates**: a blocking `CHECKLIST_TEMPLATE.md` item ("standing successor handoff current; pod lease registered; run-supervision record written") **and a blocking close-side gate ("run-supervision record cleared/inactive") beside the existing "self-wake/watchdog cleared" teardown gate** (so a finished run can't be resurrected), plus `START_TEMPLATE.md` wording telling the executor where to maintain them — the templates are where the executor actually reads its obligations, not only the SKILL prose. Ships a small **run-supervision-record helper + path/schema** (`create/update/stop/close/is-desired-active`, atomic writes) that the instance wrappers consume; the resume-contract *prose* is docs-only, but the record is real machine-consumed state with one product implementation.
2. **Pod lease at acquire + box-level reaper over it** (B.0 + B.1) — `gpu-job` acquire writes a **two-phase, deletion-scoped** lease (provisional at pod-creation: pod id + durable reaper-resolvable key ref + short default expiry; enriched after `wait_ssh` with SSH endpoint + real expiry), with the **write-failure → synchronous DELETE / emergency-record** path, **`close` only after verified provider-side deletion**, and expiry-only deletion. The standing model-free sweep reaps **only expired registered leases** and **reports unknown pods, never deletes them**. Generic lease schema + lifecycle + reaper logic in `gpu-job`; instance registry path + sweep wiring out. Closes the "agent died before arming the per-pod watchdog" hole *without* the blanket-delete hazard. The lease (B.0) lands first; the sweep (B.1) consumes it.
3. **Relaunch supervisor — crash/exit + explicit-marker first cut** (B.2 reliable half) — supervises only runs that are **desired-active with no deliberate-stop marker** per the run-supervision record (never resurrects a clean `/quit`); relaunches on a `claude` exit the in-pane loop can't catch (session/pane/wrapper dead) or an agent-written *needs-relaunch* marker; the substrate-neutral decision tree (`resume_same_session` else `launch_successor(handoff_path)`, with `handoff_path` from the run-supervision record) + single-writer lock + idempotence. Generic decision logic + the desired-state contract as an `experiment-lifecycle` reference; the concrete command mapping + systemd/bash wiring are instance. **Does NOT include silent-wedge detection.**
4. **Settings quick-wins + `StopFailure` notifier** (C) — `API_TIMEOUT_MS` / `API_FORCE_IDLE_TIMEOUT` + the fire-not-resume hook (which can also drop the child-3 *needs-relaunch* marker). Mostly instance; the "notifier ≠ recovery" note is a one-line product clarification.

This design also spawns ONE `needs-design` follow-up (its own design pass, not implemented from this doc):

5. **Silent-wedge detection** (`needs-design`) — the substrate-specific, model-free root-agent activity signal that distinguishes a long healthy model turn from a true hang (the open problem named in B.2). Filed as `needs-design`, not `ready`, because the reliable signal doesn't exist yet.

Children 1 and 2 are independent and can land first (2 closes the cost-leak directly; within it the lease + its lifecycle API precede the sweep). Child 3 depends on **child 1's run-supervision record + standing-handoff contract** (its desired-state input and the fresh-successor branch's `handoff_path`) and **links to child 2's leases** by pod id to know which pods a relaunched run owns. Child 4 is independent. Child 5 is a future design pass.

## Alternatives considered

- **One monolithic supervisor issue, build directly (skip the design PR).** Rejected: the issue is `needs-design` precisely because it conflates four asks across the product/instance seam and overlaps existing machinery. Building blind would either duplicate `claude-pane-loop.sh` or bury generic logic in instance scripts. The decomposition *is* the deliverable.
- **Make the supervisor itself model-driven (an LLM watchdog).** Rejected at the root: the whole point is surviving an API outage. A model-free probe is non-negotiable; this is why the liveness check is mtime/heartbeat-based, not "ask a model if the agent looks stuck."
- **Lean only on `claude-pane-loop.sh` (it already resumes on crash).** Rejected as incomplete: it acts on process *exit* only, so it misses the wedge (alive-but-hung) case, has no fresh-successor fallback for unrecoverable sessions, and does nothing about orphaned pods. It's the foundation to build on, not the whole answer.
- **Per-run self-destruct backstop only (the current interim).** Rejected as sufficient: the `setsid sleep + curl DELETE` backstop caps *cost* but only *reaps* — it never *resumes* the run. The supervisor's distinguishing value is resume; reaping is the half we already have.
- **Wire the supervisor to a subagent executor.** Rejected per #52: a subagent isn't independently resumable (no `--continue`), which is itself a reason the durable executor must be a `new-claude.sh` session. The supervisor assumes a real session substrate.

## Blast radius

This design PR: **`proposals/54-crash-resilience-supervisor.md` only** (doc-only, `--design` gate). No code.

- **Instance (NOT this repo):** the systemd unit / bash supervisor loop, this box's session/path/RunPod wiring under `~/.config/`, the `settings.json` quick-wins, the `claude-pane-loop.sh` liveness extension. ## Rollout + rollback

Ship each child through `ship-change` independently. Sequence by dependency: children 1, 2, 4 first (independent; 2 closes the cost-leak directly, and within it the **lease (B.0) must land before the sweep (B.1)** — the sweep is unsafe without the trusted lease to read); child 3 after child 1 lands the standing-handoff contract it consumes. Roll the **reaper sweep** out in **dry-run first** (log the DELETEs it *would* issue, reconciled against the ledger + the lease registry) before it deletes for real — a buggy reaper that kills a live run is the one new failure mode this introduces, so it gets a watched dry-run window; the **report-unknown-never-delete** rule means an un-leased peer pod is never at risk even out of dry-run. The **supervisor relaunch** keeps `claude-pane-loop.sh`'s existing crash-storm cap so a relaunch loop can't spin. Each child is a normal one-commit squash on `main`; rollback is the standard revert. Disabling the instance supervisor is a `systemctl disable` (or removing the loop) with no product change required — the product-side contract (lease schema, stuck contract, handoff) is inert without the instance supervisor consuming it.
